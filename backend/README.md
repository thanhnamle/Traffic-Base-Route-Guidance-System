# Backend

This backend is the route-guidance engine for the project. It sits between:

- prepared model predictions in `results/predictions/`
- the SCATS graph in `backend/generated/`
- the frontend, which requests routes through the local API

## What the backend does

- loads the SCATS/Boroondara graph
- looks up predictions for the selected model:
  - `lightgbm`
  - `lstm`
  - `gru`
- converts prediction values into edge travel-time costs
- runs A*
- returns top-k routes to the frontend

The backend currently **does not retrain models** and **does not run model inference directly from model files**.  
It uses prepared predictions files such as:

- `results/predictions/2006_predictions.csv`
- `results/predictions/2014_predictions.csv`

generated by the shared `main` prediction pipeline.

## Folder structure

`core/`
- `config.py`: paths for the SCATS graph and prepared predictions CSV files
- `assumptions.py`: speed limit, intersection delay, congestion scaling

`generated/`
- `scats_nodes.json`: generated SCATS nodes
- `scats_edges.json`: generated SCATS edges

`models/`
- `model_loader.py`: loads the prepared predictions CSV
- `prediction_inference.py`: builds site-level flow lookups from the selected model column

`route_guidance/`
- `types.py`: dataclasses for nodes, edges, routes, and segments
- `graph_builder.py`: loads route graphs from JSON
- `build_scats_graph.py`: generates the SCATS graph from traffic data
- `heuristic.py`: haversine distance and straight-line A* heuristic
- `travel_time.py`: converts predicted flow into estimated travel time
- `astar.py`: shortest-path search using A*
- `top_k.py`: returns up to 5 routes
- `route_formatter.py`: formats backend routes into the structure expected by the frontend

`services/`
- `route_service.py`: main orchestration layer connecting graph + prediction lookup + route search

Root files:
- `api_server.py`: local backend API used by the frontend
- `main.py`: quick local backend smoke test

## Current flow

1. `src.predict --data 2006` and `src.predict --data 2014` generate:
   - `results/predictions/2006_predictions.csv`
   - `results/predictions/2014_predictions.csv`
2. The backend loads that CSV.
3. The backend selects the correct prediction column based on `algorithm`:
   - `predicted_lightgbm`
   - `predicted_lstm`
   - `predicted_gru`
4. It aggregates movement-level predictions into site-level flow values.
5. It converts site-level flow into edge travel-time costs.
6. It runs A* + top-k route search.
7. The frontend calls `/api/routes` and renders the results.

## Local API

`GET /api/health`
- checks whether the backend is running

`GET /api/graph`
- returns the SCATS graph for the frontend

`GET /api/routes?origin=970&destination=4043&algorithm=lightgbm&data=2014&k=5`
- returns top-k routes for the selected model

Supported `algorithm` values:
- `lightgbm`
- `lstm`
- `gru`

Supported `data` values:
- `2006`
- `2014`

## How to run

Run the backend API:

```powershell
py -m backend.api_server
```

Run the quick local demo:

```powershell
py -m backend.main
```

Health check:

```text
http://127.0.0.1:8000/api/health
```

Example route request:

```text
http://127.0.0.1:8000/api/routes?origin=970&destination=4043&algorithm=lightgbm&data=2014&k=5
```

## Notes

- frontend and backend are now separated
- the backend currently uses the real SCATS graph
- top-k now returns up to 5 routes
- if the prepared predictions CSV is missing, route generation will not match the selected model
- invalid `algorithm` or `data` values now return HTTP 400 instead of silently falling back
